# n8n workflows

This directory is a backup/source-of-truth mirror of what's actually deployed on the n8n instance — n8n's own Postgres database on the VM is the live copy, this is version control for it.

## Files

- `workflows/telegram-agent.json` — the main Telegram bot workflow (routing, RAG, reminders, projects, status, CRUD). Generated by [`scripts/build_n8n_workflow.py`](../scripts/build_n8n_workflow.py) — edit the prompts in `router-prompts/` or the script itself, don't hand-edit this JSON.
- `workflows/cron-reminder-checker.json` — runs every minute, fires due reminders.
- `workflows/cron-daily-digest.json` — runs daily at 9:00 Baku time, summarizes active projects + upcoming reminders.

## Deploying a change

1. Edit the relevant prompt in `router-prompts/`, or `scripts/build_n8n_workflow.py` for structural changes.
2. Run `python3 scripts/build_n8n_workflow.py` — regenerates `workflows/telegram-agent.json`.
3. Push it to the live n8n instance via the REST API (`PUT /api/v1/workflows/{id}`) using the `N8N_API_KEY` from `.n8n_secrets`.
4. **Publish the workflow from the n8n editor UI** (Serverless-style versioned publish) — the REST API's `activate` endpoint reliably updates the `active` flag but does *not* reliably re-register the Telegram webhook. This is a known n8n quirk (see the `n8n-api-gotchas` memory) — always verify with a fresh test message after deploying, not just a green API response.

The cron workflows (`cron-reminder-checker.json`, `cron-daily-digest.json`) use plain Schedule Triggers, not webhooks, so they activate reliably via the API with no extra step.
